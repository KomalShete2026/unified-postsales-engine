2. Problem Statement & Business Context

2.1 Current State Analysis

2.1.1 Fragmented Tool Landscape

Post-sales operations at PANW currently rely on three disconnected systems:

System Purpose Users Pain Points
Salesforce (SFDC) CRM, Case Management Pre-Sales, PS Managers • No task management capabilities
• Limited visibility into PS execution
• Manual data entry required
Clarizen (Planview) Project Management PS Consultants, PS Managers • Not integrated with SFDC
• No AI-powered insights
• Manual resource allocation
Gainsight Customer Success Management CS Managers • Receives incomplete handoffs
• No visibility into PS progress
• Delayed deployment tracking

Key Problem

Three sources of truth lead to constant sync issues, context loss during handoffs, and manual rework. Consultants spend ~15 hours/month just on status updates across systems.

2.1.2 Manual Planning with Delayed Activation

Current Consultant Assignment Process:

  1. Pre-Sales submits case to PS Manager via email/Slack
  2. PS Manager reviews case manually (~2 hours)
  3. PS Manager checks consultant availability in spreadsheet
  4. PS Manager queries team via Slack for expertise
  5. PS Manager makes gut-feel decision based on limited data
  6. Assignment communicated via email/Slack
  7. Consultant manually creates tasks in Clarizen

Time to Assign: 3-5 business days (average)

Assignment Quality: 70% "Best Fit" (based on retrospective analysis)

Data Source: Analysis of 120 post-sales engagements (Jan-Dec 2025) showed that 30% of initial assignments required consultant reassignment due to skill mismatch, workload conflicts, or availability issues.

2.1.3 Context Lost in Handoffs

Pre-Sales → PS Handoff Issues:

  • SOW documents stored in SFDC but not accessible in Clarizen
  • Business goals captured in pre-sales calls but not documented
  • Customer integrations list recreated from scratch by PS consultant
  • Average 2 days lost in discovery that pre-sales already completed

PS → CS Handoff Issues:

  • Technical Requirements Document (TRD) sent via email, not tracked
  • Configuration files uploaded to shared drive, links break
  • Test results documented in Clarizen but CS doesn't have access
  • Average 2 days handoff delay due to missing deliverables

FR-PM-002: AI Task Generation

Description: PS Manager generates tasks using AI based on case context.

Acceptance Criteria:

  • Button "AI Generate Tasks" visible when status = "ps_review"
  • Click button triggers API call to /api/unified/ai/generate-tasks
  • System runs 5 AI agents in parallel:
    1. Case Context Analyzer: Analyzes ARR, products, integrations, complexity
    2. Task Decomposition Agent: Breaks down work into 5-7 discrete tasks
    3. Dependency Mapper: Identifies task dependencies (depends_on, blocks)
    4. Priority Ranker: Assigns priority (P0-P4) to each task
    5. Time Estimator: Estimates hours per task based on historical data
  • Expected task types: Discovery, Architecture, Configuration, Integration, Testing, Documentation, Training
  • Each task created with:
    • id: TASK-{case_id}-{counter}
    • ai_generated: true
    • ai_confidence: 0-1 score
    • ai_reasoning: Why this task was suggested
  • Update case status: ps_review → tasks_generated
  • Show success message: "Generated {count} tasks"
  • Latency: <3 seconds

Priority: P0 (Must Have)

FR-PM-003: Multi-Agent Consultant Recommendation

Description: PS Manager runs 25-agent AI analysis to recommend best consultant for case.

Acceptance Criteria:

  • Button "Run 25-Agent AI Recommendation" visible when status = "tasks_generated"
  • Click button triggers API call to /api/unified/ai/multi-agent-assignment
  • System executes 14 specialized agents in parallel (see section 10 for detailed agent specs):
    • Tier 1 (4 agents): Sales Intelligence, SFDC Support Risk, Implementation RAG, Scope Creep Detection
    • Tier 2 (6 agents): Skills-Based Optimizer, Calendar Availability, Workload Intelligence, Performance Analysis, Complexity Matching, Capacity Planning
    • Tier 3 (2 agents): GDPR Compliance, Multi-Cloud Orchestration
    • Tier 4 (1 agent): Sentiment Analysis
  • Orchestrator agent (Claude Opus 4.6) synthesizes all 14 outputs into final recommendation
  • Show modal with:
    • Recommended consultant name
    • Confidence score (0-100%)
    • Reasoning (2-3 sentences)
    • Key risks identified
    • Estimated timeline (days)
    • Alternative consultant option
    • Individual agent results (collapsible list)
  • PS Manager can approve or override recommendation
  • Log all agent decisions to AGENT_DECISIONS table for audit
  • Latency: <20 seconds

Priority: P0 (Must Have)

3.5 Cross-Functional Requirements (FR-CF-*)

FR-CF-001: Activity Audit Trail

Description: System logs every significant action for audit and compliance.

Acceptance Criteria:

  • Create Activity record for:
    • Case created/updated
    • Tasks generated
    • Assignment created/updated
    • Task status changed
    • Handoff created/accepted
    • Case deployed/completed
  • Each activity includes:
    • id: ACT-2026-{counter}
    • action: Short description (e.g., "Case Created")
    • description: Detailed description
    • actor: User email who performed action
    • actor_name: User full name
    • actor_role: User role
    • timestamp: ISO 8601 timestamp
    • case_id, task_id, assignment_id, handoff_id: Related entity IDs
  • Activities viewable on case details page, sorted by timestamp desc

Priority: P1 (Should Have)

FR-CF-002: Real-Time Status Synchronization

Description: Status changes in one entity automatically propagate to related entities.

Acceptance Criteria:

  • When task status changes:
    • Find assignment containing this task
    • Calculate: completed_tasks / total_tasks
    • If 100% complete → Assignment status = "completed"
    • If >0% and <100% → Assignment status = "in_progress"
    • If 0% → Assignment status = "pending"
  • When assignment status = "completed":
    • Update case status: assigned → in_progress (if not already)
  • When handoff created:
    • Update case status: in_progress → handed_off
    • Set case.handed_off_at timestamp
  • All updates logged as activities

Priority: P0 (Must Have)

FR-CF-003: Data Integrity & Cleanup

Description: System prevents duplicate records and cleans up orphaned data.

Acceptance Criteria:

  • Duplicate Prevention:
    • Before creating handoff, check if handoff exists for case_id with status != "completed"
    • If exists, return existing handoff instead of creating duplicate
    • Before creating task, check if task with same title exists for case_id
    • If exists, skip creation (log warning)
  • Orphaned Data Cleanup:
    • Assignment.task_ids should only contain IDs of tasks that exist
    • On assignment status sync, remove task IDs that don't have matching task records
    • Log: "Cleaned {count} orphaned task_ids from assignment {id}"
  • Run cleanup on server startup and on-demand via API endpoint

Priority: P1 (Should Have)
