Product Vision & Principles

Product Vision

"Enable seamless context flow from Pre-Sales → Professional Services → Customer Success, eliminating silos and ensuring every deployment has the right person, right information, at the right time."

Design Principles

1. Single Source of Truth

All personas work from the same unified database. No duplicate data entry, no orphaned records.

Why: Every handoff currently causes data loss because teams use separate systems (SFDC → Clarizen → Gainsight).

2. AI Augments, Humans Decide

AI generates recommendations, humans make final decisions. No black-box assignments.

Why: Trust is critical. Managers won't adopt if they can't understand/override AI decisions.

3. Progressive Enhancement

Core workflows work without AI. AI enhances speed and accuracy but isn't a hard dependency.

Why: Reduces risk of adoption failure if AI performance degrades.

4. Workflow-First, Not Feature-First

Design around complete end-to-end workflows, not isolated features.

Why: Solving one persona's problem without considering handoffs just moves the bottleneck.

Product Hypothesis: If we create a unified platform where context flows automatically from Pre-Sales → PS → CS, we will reduce deployment time and increase success rates without requiring teams to change their existing tools (SFDC, Gainsight, Clarizen).

Key User Stories

Epic 1: AI-Powered Case Intake

User Story 1.1: As a Pre-Sales Engineer, I want to upload an SOW and have the system auto-extract customer requirements, so that I don't spend hours manually entering data into SFDC.

Acceptance Criteria:
• User uploads PDF/DOCX SOW
• System extracts: customer name, products, integrations, timelines
• Pre-populated form displayed within 30 seconds
• User can edit any auto-filled field
• Optional SFDC sync to push data back to Salesforce

User Story 1.2: As a Pre-Sales Engineer, I want to see similar past cases when creating a new case, so that I can learn from previous deployments and avoid repeating mistakes.

Acceptance Criteria:
• System shows "Similar Cases" sidebar with 3-5 matches
• Match based on: product + industry + integration
• Show: case outcome, deployment time, key lessons learned
• User can click to view full case details

Epic 2: Intelligent Resource Allocation

User Story 2.1: As a PS Manager, I want to receive AI-recommended consultant assignments with confidence scores, so that I can make data-driven decisions instead of guessing.

Acceptance Criteria:
• AI suggests top 3 consultants ranked by fit score
• Each recommendation shows: skills match, current workload, past success rate
• Manager can accept, override, or request alternative
• System logs override reason for AI learning

User Story 2.2: As a PS Manager, I want to see consultant capacity dashboard, so that I can avoid overallocation.

Acceptance Criteria:
• Dashboard shows: consultant name, current cases, total hours allocated, availability percentage
• Color coding: green (<70% allocated), yellow (70-90%), red (>90%)
• Filterable by: geography, product expertise, availability date

Epic 3: Standardized Handoff Checklist

User Story 3.1: As a PS Consultant, I want to follow a standardized handoff checklist, so that I don't miss deliverables when handing off to CS.

Acceptance Criteria:
• Checklist auto-generated based on product (e.g., Prisma Cloud requires: TRD, network diagram, test results)
• Consultant checks off each item as uploaded
• System blocks handoff if required items missing
• CS Manager receives notification when handoff complete

Technical Feasibility Overview

High-level architecture — enough to prove it's buildable

Tech Stack

  • Frontend: Next.js 14 (App Router), TypeScript, Tailwind CSS
  • Backend: Next.js API Routes (serverless)
  • Database: PostgreSQL (Replit managed)
  • AI Layer: Claude Sonnet 4.6 (tier agents) + Opus 4.6 (orchestrator)
  • LLM Gateway: LLM Layer for model routing and fallback
  • Deployment: Replit (zero-config deployment, autoscaling)

AI Agent Architecture

25 agents across 4 tiers:

  • Tier 1 (Pre-Sales): 1 agent (Document Intelligence)
  • Tier 2 (Task Gen): 5 agents (Planning, Dependency, Estimation, Validation)
  • Tier 3 (Resource): 14 agents (Skills Matcher, Workload, Performance, etc.)
  • Tier 4 (Deployment + CS): 5 agents (Progress, Blocker, Quality, Handoff)

1 Orchestrator Agent (Claude Opus 4.6) synthesizes recommendations from tier agents

Database Schema

8 interconnected tables:

  • cases - Master case records
  • tasks - Work items per case
  • assignments - Consultant allocations
  • handoffs - PS → CS transitions
  • activities - Complete audit trail
  • sow_documents - SOW repository
  • historical_assignments - Learning data for AI
  • agent_decisions - AI audit trail

Integration Architecture

Enterprise Integrations:

  • SFDC: REST API (read cases, write progress)
  • Gainsight: API (create CS tickets on handoff)
  • Clarizen: API (sync tasks, time tracking)
  • Plainview: API (capacity planning data)

V1: Manual entry only. V2: Full bi-directional sync.

Key Product Trade-offs

Trade-off #1: Build on Replit vs. AWS

Decision: Build on Replit

Why: Faster time-to-market (no DevOps), lower cost for MVP, easier for PM to demo

Downside: May need to migrate to AWS later for enterprise scale

Mitigation: Design database schema to be portable, use standard PostgreSQL (not Replit-specific)

Trade-off #2: Custom UI vs. Embed in SFDC

Decision: Custom standalone UI

Why: More flexibility, better UX, cross-team visibility (not just SFDC users)

Downside: Users have to switch between tools (SFDC + our platform)

Mitigation: V2 will sync data back to SFDC so both systems stay in sync

Trade-off #3: Multi-Model AI vs. Single Model

Decision: Use multiple Claude models (Sonnet for tiers, Opus for orchestrator)

Why: Opus has better reasoning for complex decisions, Sonnet is cost-effective for simpler tasks

Downside: Slightly more complex AI infrastructure

Mitigation: LLM Layer handles model routing automatically

Trade-off #4: Full Automation vs. Human-in-Loop

Decision: Human-in-the-loop (managers approve AI recommendations)

Why: Builds trust, reduces adoption risk, allows for nuance (e.g., customer relationships)

Downside: Not fully automated (still requires manager review)

Mitigation: V3 may add auto-approval for low-risk cases
