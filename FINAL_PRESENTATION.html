The Problem: Why We Built This

After a deal is sold, deployment is slow & inconsistent — work passes through three disconnected teams on three disconnected systems of record, and context is lost at every handoff.

Pre-Sales
Salesforce · owns the deal & context
→
Professional Services
Clarizen · owns delivery & tasks
→
Customer Success
Gainsight · owns customer health

⚠ Pain #1 · Handoff Pre-Sales → PS

Context lives in Salesforce & Slack and is lost at handoff — PS re-discovers what was already known.

⚠ Pain #2 · Inside PS

Resourcing defaults to "First Available," not "Best Fit"; engineers recreate TRDs from scratch despite similar past work.

⚠ Pain #3 · Handoff PS → CS

Handoffs are incomplete & non-standard — missing deliverables; onboarding stalls while CS chases docs.

Root cause: the teams operate in silos — there is no single system of engagement carrying context & institutional knowledge across the journey. The result: slow time-to-value, delivery friction, and an org that never learns — translating directly to churn risk, delayed revenue, and a competitive gap on deployment speed (vs. CrowdStrike, Zscaler), at a scale manual handoffs can't survive.

Core Insight

The real problem isn't any single team's inefficiency — it's that silos prevent context from flowing from Pre-Sales → PS → CS. Every handoff is a point of failure. Solve the handoff, and you solve all four personas at once.

Solution Overview

Core Value Proposition

One system of engagement that sits on top of the systems of record (SFDC · Clarizen · Gainsight · DMS · Slack) — carrying a customer engagement through its entire post-sales lifecycle on a single unified record, with AI assisting at every decision point and a human approving every high-stakes one.

→
PS Consultant
Execute tasks → hand off
→

⟶ ONE unified database — no re-entry, no lost context · outcomes feed a learning loop ⟲

Key Components

1. Unified Record

Single source of truth (8 tables, full audit trail) — context captured once, never re-entered across personas.

2. AI Across the Lifecycle

Briefing → task generation → best-fit assignment → customer health (the 4 AI use cases that follow).

3. Best-Fit Resourcing

Transparent weighted scoring — skills & familiarity over raw availability; "First Available → Best Fit."

4. Enterprise Integrations

Bidirectional, loop-safe sync with SFDC · Clarizen · Gainsight — single-writer ownership, origin-stamped.

5. Customer Health & Outcomes

Health score, escalation risk, sentiment & Time-to-Value — the "Outcome Engine" payoff on the CS side.

6. Standardized CS Handoff

Complete deliverables package (TRD, configs, test results) enforced before PS → CS handoff.

7. HITL + Learning Loop

Humans gate every high-stakes decision; every completed outcome improves the next recommendation.

AI Use Case 1 — Case Briefing & Summary

👤 Persona: PS Manager (on "View Details") 🧠 Method: LLM summarization & synthesis Model: fast, low-cost LLM (Haiku-class / mini)

What Happens

When the PS Manager opens a case that just arrived from Pre-Sales, instead of reading the whole record they get an instant AI brief — executive summary, top risks, recommended actions, engagement sentiment, and communication highlights — synthesized from the case data, activity trail, and communications.

Why It Matters

The direct antidote to "context lost at handoff." The manager understands the case in seconds (not 20 minutes), gets the same quality brief every time, and sees risks surfaced proactively — so the triage & assignment decision is better.

AI Use Case 3 — Best-Fit Resource Allocation

👤 Persona: PS Manager 🧠 Method: deterministic scoring + multi-agent context + synthesis Model: rules/math · Sonnet agents · Opus synthesizer · RAG (history) 🤖 Agents: 13 (Tier 1–4) → Orchestrator

How it works — three layers

Layer 1 · Deterministic weighted scoring (the actual match — plain math, not AI)
Skills match35%
Recency20%
Customer familiarity15%
Timezone10%
Availability10%
Utilization10%
Layer 2 · Agent context (13 agents run in parallel)

Tier 1: Sales Intelligence · SFDC Support Risk · Implementation-RAG · Scope Creep
Tier 2: Skills · Workload · Performance · Complexity · Calendar · Capacity
Tier 3: GDPR · Multi-Cloud  ·  Tier 4: Sentiment

Implementation-RAG retrieves similar past engagements to ground complexity/timeline & familiarity.

Layer 3 · Orchestrator → Human

Synthesizes score + context → recommendation + reasoning + confidence (lowers confidence on disagreement) → PS Manager approves / overrides / splits across a team.

What Happens

The score ranks consultants; agents add context (incl. RAG over past cases); the Orchestrator explains the pick — then the manager approves / overrides / splits. The match is deterministic math, not an LLM — so it's fair, reproducible, and a manager can defend it.

Why It Matters

The shift from "First Available" → "Best Fit." Availability is only 30% combined; skill + recency + familiarity are 70% — mathematically why it's Best-Fit.

Evaluation

Implicit signal — what the manager does with the pick (diagnostic):

✅ Approvedaccepted as-is — the headline (good)
🔄 Overrode→ wrong pick / missing context
↩️ Reassigned mid-stream→ true bad fit (strongest negative)

Backtest: match-score ↔ actual outcome correlation (would the score have picked the consultant who succeeded?).

Outcomes

⭐ First-Available → Best-Fit lift Fewer reassignments ↓ Time-to-assign Auditable, defensible picks Burnout guarded

Ladders up to the North Star: Best-Fit score (right person) → Approval (manager accepts) → ↑ Deployment success + fewer reassignments → ⭐ Time-to-Value (Deal Close → Deployment Success)

AI Use Case 4 — Customer Health Signals

👤 Persona: CS Manager (surfaced on all dashboards) 🧠 Method: deterministic score + LLM sentiment Model: rules/math · reasoning LLM (Sonnet-class) + heuristic fallback 🤖 Agents: Health Scoring · Sentiment · Escalation Risk (Tier 4)

How it works — two layers

Layer 1 · Deterministic health score (0–100 + A–F grade — plain math)

Starts at 100, deducts for real signals: SLA overrun (−35) · blocked tasks (−25) · low completion · missing docs (−12). Auditable & reproducible.

Layer 2 · LLM sentiment / risk

Reads emails · Slack · notes → sentiment (positive→concerning) · risk signal · narrative · a 0–20 penalty that adjusts the score. Keyword fallback if the call fails.

Then → proactive action

Health < 60 → Escalation Risk Predictor fires (outreach + manager alert). Outcome + Time-to-Value feed the learning loop.

What Happens

Every customer gets a continuous 0–100 health score from real delivery signals, nudged by an LLM read of tone & risk in their comms — surfacing at-risk accounts before they churn. Score is deterministic (auditable "C grade"); sentiment needs an LLM (tone isn't keyword-matching).

Why It Matters

The "Outcome Engine" payoff & the assignment's "telemetry-guided optimization": proactive, measures Time-to-Value, and feeds the learning loop so every deployment makes the next recommendation smarter.

Evaluation

🎯 Precisiondid flagged accounts actually escalate? (not false alarms)
🔔 Recalldid we catch the ones that did churn?
⏱️ Lead timeflagged before the escalation, not after
🙂 Sentiment accuracyvs. a human-labeled set

+ CS Manager ✅ acted / 🚫 dismissed the alert → feeds the learning loop.

Outcomes

⭐ Churn / escalations caught early ↓ Time-to-Value Proactive, not reactive CS Closes the learning loop

Ladders up to the North Star: Health signal (accurate) → Early escalation flag (acted on) → ↓ churn + ↓ Time-to-Value → ⭐ Time-to-Value (Deal Close → Deployment Success)

DMS SOW · Slack conversation · SFDC deal/$$ · Clarizen delivery & hours · Gainsight health/NPS · Engine case state. One writer per field → no sync loops.

Loop-prevention contract

  1. Every outbound write is origin-stamped + carries a correlationId.
  2. Inbound webhook with a matching id = our own echo → dropped.
  3. One writer per field → last-writer-wins, no conflicts.
  4. Every sync action logged to the audit trail.
Why it matters: the engine is a system of engagement on top of the systems of record — it never owns what they own, so sync is bidirectional but provably loop-safe and auditable. (Demo: the /sync-status "fire inbound webhook" button shows the drop live.)

Data Architecture

Contract-first: design the data, then build the AI & integrations around it — the schema is the spec

① Data structure — the unified record

cases (master record)
 ├─ tasks · assignments · handoffs
 ├─ activities (audit) · agent_decisions
 └─ historical_assignments · performance

8 connected tables · every child references case_id → context never fragments.

Provenance per field

{ arr: { value:850000, source:"sfdc" },
  timeline: { value:"Jul 15", source:"slack" } }

Each field records where it came from → proves "no lost context," fully auditable.

② Data flow / pipeline

Capture (Deal ID → fetch SFDC + SOW)
  ↓ Structure → typed unified case record
  ↓ Enrich → AI tasks · scores · health (write back to record)
  ↓ Sync → SFDC · Clarizen · Gainsight (single-writer)
  ↓ Learn → outcomes → historical_assignments + performance
  ⟲ feeds the next recommendation

③ Why this approach

  • One unified record, not per-team DBs → the architectural fix for the silo problem; context never re-entered.
  • Contract-first → design the data model, build AI around it (testable, typed).
  • Provenance + audit trail → every value & AI decision traceable.
  • Prod target: Postgres + pgvector → relational truth + vector search for RAG; in-memory now, schema is production-shaped (mechanical migration).

④ Data strategy

Synthetic but use-case-aligned seed (5 cases spanning health A–F, 8 SOWs). At scale: continuously ingest completed engagements + product docs into the vector store → powers RAG & the learning loop.

Appendix · Principles & Build vs. Buy

Design Principles

1. Single Source of Truth

All personas work from one unified record. No duplicate entry, no orphaned data.

2. AI Augments, Humans Decide

AI recommends with transparent reasoning; humans gate every high-stakes call.

3. Grounding, Not Guessing

RAG + mandatory citations + a critic gate keep AI output technically accurate.

4. Workflow-First

Design around the end-to-end journey, not isolated features — or you just move the bottleneck.

Build vs. Buy

Decision Chosen Why
Platform Build a unified AI-native system of engagement No vendor solves silos + automation while integrating with SFDC/Clarizen/Gainsight
Model gateway Reuse LiteLLM gateway OpenAI-compatible, centralized cost/keys, instant model swap
Vector DB Buy (Pinecone / pgvector) Retrieval is solved infra — build the grounding logic, not the index
Models Opus (reason) + Sonnet (volume) + Haiku (classify) Match capability to task; pay for reasoning only where it changes the outcome
