The Problem: Why We Built This

Post-sales delivery runs across three disconnected systems of record — Salesforce (the deal), Clarizen (PS delivery), and Gainsight (CS health). Context dies at every handoff between Pre-Sales → Professional Services → Customer Success.

Context is lost at handoff

Pre-sales requirements live in Salesforce and don't carry forward. The receiving team re-discovers what was already known, so work gets re-done and kickoffs slip.

Engineers recreate designs from scratch

HLDs and TRDs are rebuilt for each engagement even when many similar past deployments already exist — there is no reliable way to reuse that knowledge.

Assignment defaults to "First Available"

Consultants are matched on availability and gut feel rather than skills, recency, and customer familiarity — hurting deployment quality and time-to-value.

Handoffs are inconsistent & incomplete

No standardized deliverables checklist between PS and CS, so packages arrive missing artifacts and onboarding stalls while teams chase documentation.

Core Insight

The real problem isn't any single team's inefficiency — it's that silos prevent context from flowing from Pre-Sales → PS → CS. Every handoff is a point of failure. Solve the handoff, and you solve all four personas at once.

AI Use Case 1 — Intelligent Case Intake

Turn unstructured pre-sales context into a structured Design of Record

🧠 Method: LLM structured extraction (document → typed JSON) Model: reasoning LLM (Opus-class) + schema-constrained output Why: long, messy SOWs need strong long-context reasoning to extract accurately

The job

Capture customer requirements once, in a structured form, so nothing is re-keyed or lost as the case moves downstream.

How the AI helps

  • Reads an SOW (or a fetched SFDC case) and extracts the key fields — customer, ARR, products, integrations, timeline, business goals.
  • Maps them into a strongly-typed Design of Record (DoR) — the AI's input contract for every downstream step.
  • The engineer reviews & confirms; data_source is stamped sfdc or manual for provenance.

Flow

SOW / SFDC case (unstructured)
  ↓
Extract: customer · products · integrations · timeline
  ↓
Map → Design of Record (typed contract)
  ↓
Human review & confirm
  ↓
Create unified case · status submitted

Why a DoR first: design the data contract, then build the AI around it — the schema is the spec.

Why it matters: this is the bridge that ends re-entry and lost context — captured once at intake, every downstream agent reads the same structured record.

AI Use Case 2 — Case Briefing & Summary

A persona-aware briefing so context is never re-discovered at a handoff

🧠 Method: LLM summarization & synthesis (context → JSON brief) Model: fast, low-cost LLM (Haiku-class / mini) Why: short, frequent, low-stakes summaries — optimize for speed & cost, not deep reasoning

The job

When a case lands on someone's desk — a PS Manager receiving it, a Consultant being assigned — give them an instant, plain-English brief instead of making them re-read the whole record.

How the AI helps

  • Synthesizes the case data, activity trail, document status, and communications into a 2–3 sentence executive summary.
  • Surfaces top risks, recommended next actions, a timeline read, and an engagement sentiment (on-track / needs-attention / at-risk).
  • Persona-aware: the PS Manager gets a triage/assign view; the Consultant gets an execution head-start view.

Output

  • Executive summary
  • Key risks (with detail)
  • Recommended actions
  • Engagement sentiment + timeline assessment
  • Communication highlights

Best-fit model: a fast, cost-effective model (e.g. a Haiku-class / mini model) — structured JSON, high volume, low latency.

Why it matters: this is the direct antidote to "context lost at handoff" — every receiving persona starts with the same AI-synthesized picture instead of re-discovering it.

AI Use Case 3 — Task Generation

From a structured Design of Record to a standardized task list

🧠 Method: LLM structured generation (DoR → task JSON) + template fallback Model: high-volume workhorse LLM (Sonnet-class) Why: best price/performance for frequent structured generation; reliable JSON

The job

Break a case down into the right deployment tasks — consistently, regardless of which manager or product line it is.

How the AI helps

  • Reads the case (products, business goals, integrations) and generates 5–8 lifecycle tasks: discovery → architecture → configuration → integration → testing → documentation → training.
  • Each task carries type, priority, hour estimate, dependencies, and an AI confidence score + reasoning.
  • Robustness built in: 30s timeout, duplicate-prevention, and a deterministic template fallback so it never dead-ends.

Flow

Case (Design of Record)
  ↓
Build prompt: products · goals · integrations
  ↓
Claude Sonnet → JSON task list
  ↓
Persist tasks (ai_generated, confidence, depends_on)
  ↓
Case status → tasks_generated

Model: claude-sonnet-4-6 · best price/performance for high-volume structured work.

Why it matters: this is the mechanical bridge from pre-sales metadata → post-sales work — the standardization every PS Consultant said was missing.

AI Use Case 4 — Best-Fit Resource Allocation

The shift from "First Available" to "Best Fit"

🧠 Method: weighted-scoring rubric + multi-agent synthesis (LLM ranks top candidates) Model: Sonnet workers → Opus synthesizer Why: deterministic math for fairness/auditability; LLM only to reason over ties & explain

The job

Match the right consultant to a case using a transparent, multi-dimensional score — not gut feel.

The weighted rubric

Skills match35%
Recency (worked on this tech recently)20%
Customer familiarity15%
Timezone alignment10%
Availability10%
Utilization (guards against burnout)10%

Why this is "Best Fit"

Availability is only 30% combined (availability + timezone); skill, recency and familiarity are 70%.

That weighting is mathematically why the engine produces Best-Fit, not First-Available.

Output

Top-ranked consultant with a fit score, the reasoning behind it, and a confidence value — all surfaced for the manager to approve, edit, or override.

Directly answers the "availability vs. technical familiarity" tension: familiarity and skill win, but availability and utilization keep recommendations realistic and humane.

AI Use Case 5 — TRD Generation (RAG-Grounded)

No more rebuilding designs from scratch — a cited, review-ready TRD in one click

🧠 Method: agentic RAG — Plan → Draft → Critic → Revise, with [DOC-X] citations Model: Opus (plan/critic) + Sonnet (parallel drafters) Why: hardest reasoning task — needs grounding + a tiered model split for accuracy vs. cost
1 · PLAN
Opus decides 10–12 sections + a retrieval query per section
→
2 · DRAFT
Sonnet drafts sections in parallel, each grounded in retrieved docs
→
3 · CRITIC
Reviews for unsupported claims & compliance gaps
→
4 · REVISE
Re-drafts high-severity flagged sections

RAG keeps it accurate

Each section retrieves product docs + past TRDs and injects them as [DOC-X] context. Every claim must cite a source; the critic rejects uncited claims. Grounding is enforced, not hoped for.

Human-in-the-loop by design

High-stake sections (Scope, Pricing, Timeline, Success Criteria, Migration) are tagged stake: High and require human approval before the TRD ships.

Tiered models: Opus for the hard reasoning (planning, synthesis); Sonnet for high-volume drafting and the critic. We pay for reasoning only where it changes the outcome.

AI Use Case 6 — Customer Health Signals

Telemetry-guided optimization — read sentiment and surface risk on the CS side

🧠 Method: LLM sentiment/risk analysis + keyword-heuristic fallback Model: reasoning LLM (Sonnet-class) Why: nuanced tone needs an LLM; heuristic fallback guarantees a signal if the call fails

The job

Turn unstructured signals — emails, Slack threads, notes — into an early-warning read on customer health, so risk is caught before it becomes churn.

How the AI helps

  • Analyzes communication sentiment and pre-sales risk signals into an AI health narrative.
  • Feeds a health score, NPS, adoption, and renewal-risk view on the Customer Success dashboard.
  • Closes the loop: the outcome + sentiment flow into the learning loop to improve future best-fit recommendations.

Flow

Emails · Slack · notes (unstructured)
  ↓
Sentiment + risk analysis
  ↓
Health score · NPS · renewal risk · narrative
  ↓
CS dashboard + learning loop

Best-fit model: a strong reasoning model for nuanced sentiment + a keyword-heuristic fallback for resilience.

Why it matters: this is the "Outcome Engine" payoff — measuring Time-to-Value and customer health, and feeding it back so the system gets smarter every cycle.

Agent Spotlight — What & Why

For key agents: the use case it performs, and why a dedicated agent (vs. one big prompt)

Agent (tier) Use case it performs Why a dedicated agent
Implementation RAG (T1) Searches historical cases for similar deployments to estimate timeline & complexity Grounds estimates in real past engagements — RAG keeps it factual, not guessed
Skills-Based Optimizer (T2) Multi-dimensional consultant match: skills × recency × familiarity × timezone Isolated, testable scoring logic — the core of "Best Fit," independently tunable
Workload & Burnout (T2) Caps concurrent load (max ~3 cases) to prevent over-allocation A single guardrail concern — keep it separate so it can veto humanely
GDPR / Compliance (T3) Flags data-residency & regulated-industry requirements (EU, SOC2, HIPAA) Compliance is high-stakes & rule-bound — a focused agent makes it auditable

Escalation Risk Predictor 🆕 (T4) Predicts escalation likelihood from health-score trend, priority, ARR & engagement signals Triggers when health < 60 → proactive outreach + manager alert; a predictive job distinct from scoring

Why agents at all? Each is a small, single-responsibility (input) → result function returning confidence, latency & cost — independently testable, observable, swappable, and resilient (one failure degrades gracefully). That's not achievable with one monolithic prompt.

End-to-End Workflow Journey

Step 1 · Pre-Sales — Create the case

Action: Pre-Sales creates a case from a SOW (repository) or an SFDC fetch.

The unstructured SOW + SFDC context becomes a structured Design of Record.

Outcome

Context is captured once, structured, and never re-entered downstream.

Step 2 · PS Manager — AI tasks + best-fit assignment

Action: Manager clicks Generate Tasks, then runs Multi-Agent Assignment.

AI produces a standardized task list, then the orchestrator fans out 5 agents and synthesizes a best-fit engineer + priority + timeline + risks.

Outcome

The shift from First Available → Best Fit, with the AI's reasoning fully transparent.

Step 3 · Human-in-the-Loop — approve the high-stakes call

Action: Manager opens the HITL Queue and sees the recommendation, confidence, and agent reasoning.

They approve, edit, or reject — edits are captured as structured feedback.

Outcome

AI recommends, human decides. Every high-stake decision is gated and auditable.

Step 4 · PS Consultant — execute + generate the TRD

Action: Consultant works the tasks and triggers TRD generation (Plan → Draft → Critic → Revise).

The TRD is grounded in retrieved docs with [DOC-X] citations.

Outcome

No recreating HLDs from scratch — a cited, review-ready TRD comes out of the engine.

Step 5 · CS Team — accept handoff, deploy, close the loop

Action: CS accepts the handoff with deliverables (TRD, configs, test results), then deploys.

On completion, the outcome is recorded and the consultant's performance history updates.

Outcome

Every completed case makes the next recommendation smarter — continuous improvement.

Key insight: orchestrating the whole journey on one record eliminates the handoff points where context is lost — Pre-Sales → PS Manager → PS Consultant → CS.

Appendix · Principles & Build vs. Buy

Design Principles

1. Single Source of Truth

All personas work from one unified record. No duplicate entry, no orphaned data.

2. AI Augments, Humans Decide

AI recommends with transparent reasoning; humans gate every high-stakes call.

3. Grounding, Not Guessing

RAG + mandatory citations + a critic gate keep AI output technically accurate.

4. Workflow-First

Design around the end-to-end journey, not isolated features — or you just move the bottleneck.
