The Problem: Why We Built This

After a deal is sold, deployment is slow & inconsistent — work passes through three disconnected teams on three disconnected systems of record, and context is lost at every handoff.

Pre-Sales
Salesforce · owns the deal & context
→
Professional Services
Clarizen · owns delivery & tasks
→
Customer Success
Gainsight · owns customer health

⚠ Pain #1 · Handoff Pre-Sales → PS

Context lives in Salesforce & Slack and is lost at handoff — PS re-discovers what was already known.

⚠ Pain #2 · Inside PS

Resourcing defaults to "First Available," not "Best Fit"; engineers recreate TRDs from scratch despite similar past work.

⚠ Pain #3 · Handoff PS → CS

Handoffs are incomplete & non-standard — missing deliverables; onboarding stalls while CS chases docs.

Root cause: the teams operate in silos — there is no single system of engagement carrying context & institutional knowledge across the journey. The result: slow time-to-value, delivery friction, and an org that never learns from its own deployments.

Core Insight

The real problem isn't any single team's inefficiency — it's that silos prevent context from flowing from Pre-Sales → PS → CS. Every handoff is a point of failure. Solve the handoff, and you solve all four personas at once.

Solution Overview

Core Value Proposition

One system of engagement that sits on top of the three systems of record — carrying a customer engagement through its entire post-sales lifecycle on a single unified record, with AI assisting at every decision point and a human approving every high-stakes one.

Key Components

1. Unified Record

Single source of truth (8 tables, full audit trail) — context captured once, never re-entered across personas.

2. AI Across the Lifecycle

Briefing → task generation → best-fit assignment → customer health (the 4 AI use cases that follow); TRD generation designed as a roadmap capability.

3. Best-Fit Resourcing

Transparent weighted scoring — skills & familiarity over raw availability; "First Available → Best Fit."

4. Enterprise Integrations

Bidirectional, loop-safe sync with SFDC · Clarizen · Gainsight — single-writer ownership, origin-stamped.

5. Customer Health & Outcomes

Health score, escalation risk, sentiment & Time-to-Value — the "Outcome Engine" payoff on the CS side.

6. Standardized CS Handoff

Complete deliverables package (TRD, configs, test results) enforced before PS → CS handoff.

7. HITL + Learning Loop

Humans gate every high-stakes decision; every completed outcome improves the next recommendation.

AI Use Case 1 — Case Briefing & Summary

👤 Persona: PS Manager (on "View Details") 🧠 Method: LLM summarization & synthesis Model: fast, low-cost LLM (Haiku-class / mini) 🤖 Agents: direct LLM call (no agent orchestration)

What Happens

When the PS Manager opens a case that just arrived from Pre-Sales, instead of reading the whole record they get an instant AI brief — executive summary, top risks, recommended actions, engagement sentiment, and communication highlights — synthesized from the case data, activity trail, and communications.

Why It Matters

The direct antidote to "context lost at handoff." The manager understands the case in seconds (not 20 minutes), gets the same quality brief every time, and sees risks surfaced proactively — so the triage & assignment decision is better.

Flow

PS Manager clicks View Details
  ↓
Gather: case data · activity trail · comms
  ↓
LLM synthesizes → structured brief
  ↓
Show: summary · risks · actions · sentiment

Why this model

High-frequency, low-stakes summarization the manager still reviews — so optimize for speed & cost, not deep reasoning. (Heavy reasoning is reserved for the TRD.) An LLM is needed because fluent, risk-aware summarization is language generation, not a template.

AI Use Case 2 — Task Generation (RAG-Grounded)

👤 Persona: PS Manager 🧠 Method: RAG + LLM structured generation Model: embeddings (retrieval) + workhorse LLM (Sonnet-class) 🤖 Agents: Implementation-RAG (retrieval) → Task-Generation agent

What Happens

From the case, the system retrieves similar past engagements — same products, integrations, industry — and generates 5–8 lifecycle tasks (discovery → architecture → configuration → integration → testing → documentation → training) grounded in what actually worked before, including the blockers those cases hit. Each task carries type, priority, hour estimate, dependencies, and AI confidence.

Why It Matters

Two problems solved at once: engineers stop recreating plans from scratch, and the org finally learns from its own deployments. The task list reflects PANW's proven playbook for this exact kind of deployment — not the model's generic guess.

Flow

Case (Design of Record)
  ↓
RETRIEVE similar past cases (vector search)
  · task lists · lessons · blockers
  ↓
LLM generates tasks grounded in that history
  ↓
Persist (ai_generated · confidence · depends_on)
  ↓
Case status → tasks_generated

Why this model

Two parts: finding similar cases is a semantic similarity problem → embeddings + vector search over historical engagements. Generating the structured task JSON is frequent, structured work → a workhorse LLM (best price/performance). Deterministic template fallback so it never dead-ends.

Knowledge base: historical engagements + lessons_learned / issues_encountered / best_practices. (Distinct from the TRD's RAG, which retrieves product docs — same technique, different knowledge base.)

AI Use Case 3 — Best-Fit Resource Allocation

👤 Persona: PS Manager 🧠 Method: deterministic scoring + multi-agent context + synthesis Model: rules/math · Sonnet agents · Opus synthesizer · RAG (history) 🤖 Agents: 13 (Tier 1–4) → Orchestrator

How it works — three layers

Layer 1 · Deterministic weighted scoring (the actual match — plain math, not AI)
Skills match35%
Recency20%
Customer familiarity15%
Timezone10%
Availability10%
Utilization10%
Layer 2 · Agent context (13 agents run in parallel)

Tier 1: Sales Intelligence · SFDC Support Risk · Implementation-RAG · Scope Creep
Tier 2: Skills · Workload · Performance · Complexity · Calendar · Capacity
Tier 3: GDPR · Multi-Cloud  ·  Tier 4: Sentiment

Implementation-RAG retrieves similar past engagements to ground complexity/timeline & familiarity.

Layer 3 · Orchestrator → Human

Synthesizes score + context → recommendation + reasoning + confidence (lowers confidence on disagreement) → PS Manager approves / overrides / splits across a team.

What Happens

The score ranks consultants; agents add context (including RAG over past cases); the Orchestrator explains the pick — then the manager decides.

Why It Matters

The shift from "First Available" → "Best Fit." Availability is only 30% combined; skill + recency + familiarity are 70% — mathematically why it's Best-Fit.

Why this model

The match is deterministic math, not an LLM — so it's fair, reproducible, and a manager can defend it. The LLM/agents are used only for the judgment layer (context + explanation), and RAG grounds the history signals. Right tool per job.

RAG's role here is supporting (enriches inputs), unlike Task Generation & TRD where it's central. The 25-agent catalog is the full system; a given assignment runs the relevant subset.

AI Use Case 4 — Customer Health Signals

👤 Persona: CS Manager (surfaced on all dashboards) 🧠 Method: deterministic score + LLM sentiment Model: rules/math · reasoning LLM (Sonnet-class) + heuristic fallback 🤖 Agents: Health Scoring · Sentiment · Escalation Risk (Tier 4)

How it works — two layers

Layer 1 · Deterministic health score (0–100 + A–F grade — plain math)

Starts at 100, deducts for real signals: SLA overrun (−35) · blocked tasks (−25) · low completion · missing docs (−12). Auditable & reproducible.

Layer 2 · LLM sentiment / risk

Reads emails · Slack · notes → sentiment (positive→concerning) · risk signal · narrative · a 0–20 penalty that adjusts the score. Keyword fallback if the call fails.

Then → proactive action

Health < 60 → Escalation Risk Predictor fires (outreach + manager alert). Outcome + Time-to-Value feed the learning loop.

What Happens

Every customer gets a continuous 0–100 health score from real delivery signals, nudged by an LLM read of tone & risk in their comms — surfacing at-risk accounts before they churn.

Why It Matters

The "Outcome Engine" payoff & the assignment's "telemetry-guided optimization": it's proactive, it measures Time-to-Value, and it feeds the learning loop so every deployment makes the next best-fit recommendation smarter.

Why this model

The score is deterministic math (auditable — you can explain a "C grade"). The sentiment read needs an LLM — tone & implied risk ("budget freeze," "exec disengagement") aren't keyword matches. Heuristic fallback guarantees a signal. Right tool per job.

End-to-End Workflow Journey

One case, one record — from Deal ID to deployed outcome

1
Pre-Sales — Identify & hand off no AI
Enters the Deal ID → system fetches the deal + SOW and validates completeness → hands the case to the PS Manager. → Context captured once, never re-entered.
2
PS Manager — Brief, generate tasks, assign best-fit
Gets an AI brief, clicks Generate Tasks (RAG-grounded), runs Best-Fit Allocation (deterministic score + agents → orchestrator). → "First Available → Best Fit," reasoning transparent.
3
Human-in-the-Loop — approve the high-stakes call
Manager sees the recommendation + confidence + agent reasoning and approves, overrides, or splits across a team. → AI recommends, human decides — gated & auditable.
4
PS Consultant — execute & hand off
Works the assigned tasks, tracks progress/blockers, then packages deliverables and hands off to CS. (TRD auto-generation = roadmap.) → Standardized, complete handoff.
5
CS Team — accept, deploy, close the loop
Accepts the handoff, deploys, and on completion the outcome + health + Time-to-Value are recorded → feed the learning loop. → Every case makes the next recommendation smarter.
Key insight: orchestrating the whole journey on one record eliminates the handoff points where context is lost — Pre-Sales → PS Manager → PS Consultant → CS.

Why a unified DB

Every persona reads/writes the same record — the architectural answer to the SFDC / Clarizen / Gainsight silo problem. Context is never lost at a handoff.

Why a model gateway

One endpoint for all models → swap Opus ↔ Sonnet ↔ Haiku without touching agent code. Pay for reasoning only where it changes the outcome; centralized cost & keys.

Appendix · Principles & Build vs. Buy

Design Principles

1. Single Source of Truth

All personas work from one unified record. No duplicate entry, no orphaned data.

2. AI Augments, Humans Decide

AI recommends with transparent reasoning; humans gate every high-stakes call.

3. Grounding, Not Guessing

RAG + mandatory citations + a critic gate keep AI output technically accurate.

4. Workflow-First

Design around the end-to-end journey, not isolated features — or you just move the bottleneck.

Build vs. Buy

Decision Chosen Why
Platform Build a unified AI-native system of engagement No vendor solves silos + automation while integrating with SFDC/Clarizen/Gainsight
Model gateway Reuse LiteLLM gateway OpenAI-compatible, centralized cost/keys, instant model swap
Vector DB Buy (Pinecone / pgvector) Retrieval is solved infra — build the grounding logic, not the index
Models Opus (reason) + Sonnet (volume) + Haiku (classify) Match capability to task; pay for reasoning only where it changes the outcome

Appendix · TRD Generation (RAG-Grounded)

👤 Persona: PS Consultant 🧠 Method: agentic RAG — Plan → Draft → Critic → Revise, [DOC-X] citations Model: Opus (plan/critic) + Sonnet (parallel drafters) 🔭 Designed — not yet wired
1 · PLAN
Opus decides 10–12 sections + a retrieval query per section
→
2 · DRAFT
Sonnet drafts sections in parallel, each grounded in retrieved docs
→
3 · CRITIC
Reviews for unsupported claims & compliance gaps
→
4 · REVISE
Re-drafts high-severity flagged sections

RAG keeps it accurate

Each section retrieves product docs + past TRDs and injects them as [DOC-X] context. Every claim must cite a source; the critic rejects uncited claims. Grounding is enforced, not hoped for.

Human-in-the-loop by design

High-stake sections (Scope, Pricing, Timeline, Success Criteria, Migration) are tagged stake: High and require human approval before the TRD ships.

Status & honesty: this is the assignment's headline capability and it's fully designed (the Plan→Draft→Critic→Revise contract + enforced citations). In the prototype I prioritized wiring the upstream flow end-to-end; productizing TRD = wire the route + a real vector store (pgvector/Pinecone) + activate the revise step. The contract is the deliverable; the wiring is the next sprint.
